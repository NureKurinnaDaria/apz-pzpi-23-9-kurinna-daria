2 ЗАВДАННЯ
     Метою роботи є дослідження архітектури складної програмної системи та аналіз застосованих архітектурних підходів.
     У межах практичного заняття необхідно підготувати доповідь про архітектуру однієї з відомих складних програмних систем. У даній роботі для аналізу обрано хмарну платформу Amazon Web Services (AWS) – одну з найбільших і найпоширеніших хмарних інфраструктур у світі.
     У межах роботи необхідно:
- виконати аналіз архітектури платформи AWS;
- визначити архітектурний стиль та принципи побудови системи;
- описати основні компоненти та сервіси;
- проаналізувати взаємодію підсистем та потоки даних;
- оцінити масштабованість та переваги архітектурних рішень;
- навести приклади програмної реалізації архітектурних принципів AWS;
- підготувати презентацію та відеодоповідь.

3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про систему
     Amazon Web Services (AWS) – це хмарна платформа компанії Amazon.com, Inc. (США), яка з 2006 року надає комплексні хмарні послуги через мережу Інтернет. Хмарні обчислення забезпечують гнучкий доступ до ресурсів без необхідності утримувати власну серверну інфраструктуру, що дозволяє підприємствам скорочувати витрати та швидше впроваджувати нові рішення. Сьогодні AWS є лідером ринку хмарних послуг і обслуговує мільйони клієнтів – від стартапів до великих корпорацій та урядових організацій.
     Платформа підтримує моделі IaaS, PaaS та SaaS і охоплює понад 200 сервісів, згрупованих за призначенням. AWS використовується для:
- хостингу та масштабування вебзастосунків і мікросервісів;
- зберігання та обробки великих обсягів даних;
- машинного навчання та штучного інтелекту;
- організації корпоративних мереж та безпеки;
- розробки та розгортання програмного забезпечення (DevOps, CI/CD);
- забезпечення глобальної доступності через розподілену мережу центрів обробки даних.
     Інфраструктура AWS охоплює понад 30 географічних регіонів, кожен з яких складається з кількох зон доступності (Availability Zones) – фізично розділених центрів обробки даних. Така організація забезпечує високу доступність і відмовостійкість сервісів. Архітектура платформи побудована так, щоб сервіси працювали незалежно один від одного та могли легко масштабуватися відповідно до потреб користувача.
     3.2 Архітектурний стиль системи
     Платформа AWS побудована на основі сервісно-орієнтованої архітектури (SOA) з елементами мікросервісного підходу. Кожен сервіс AWS є незалежним, слабозв'язаним компонентом, що надає чітко визначений набір функцій через стандартизовані API.
     Основними архітектурними принципами AWS є:
- слабке зв'язування (Loose Coupling) – сервіси взаємодіють через добре визначені інтерфейси та можуть бути замінені або оновлені незалежно один від одного;
- горизонтальне масштабування (Horizontal Scaling) – замість вертикального нарощування потужності одного сервера використовується додавання нових екземплярів;
- відмовостійкість (Fault Tolerance) – система спроєктована так, щоб відмова окремих компонентів не призводила до відмови всієї платформи;
- еластичність (Elasticity) – ресурси автоматично виділяються та вивільняються залежно від поточного навантаження;
- безпека як основа (Security by Design) – механізми безпеки інтегровані на всіх рівнях архітектури;
- модель спільної відповідальності (Shared Responsibility Model) – чіткий розподіл зон відповідальності між AWS та клієнтом.
     Крім того, AWS дотримується власного набору рекомендацій – Well-Architected Framework, який охоплює такі аспекти як надійність, безпека, продуктивність та контроль витрат.
     3.3 Основні компоненти системи
     Платформа AWS об’єднує велику кількість сервісів, які можна умовно поділити на кілька основних груп залежно від їхнього призначення.
     До першої групи належать обчислювальні сервіси. Основним із них є Amazon EC2, який дозволяє запускати віртуальні сервери з різними характеристиками залежно від потреб користувача. Для виконання коду без налаштування та підтримки серверної інфраструктури використовується AWS Lambda. Якщо ж система побудована на контейнерах, для їх розгортання та керування застосовуються Amazon ECS і Amazon EKS.
     Наступну важливу групу становлять сервіси зберігання даних. Зокрема, Amazon S3 використовується для зберігання файлів, документів, резервних копій та інших типів даних у масштабованому хмарному сховищі. Amazon EBS надає дисковий простір для віртуальних серверів, а Amazon EFS – спільне файлове сховище для кількох сервісів.
     Окрему категорію формують сервіси баз даних. AWS надає можливість працювати як із реляційними, так і з нереляційними базами даних. Наприклад, Amazon RDS призначений для керованих реляційних баз даних, Amazon DynamoDB – для високопродуктивних NoSQL-рішень, Amazon Redshift використовується для аналізу великих масивів даних, а Amazon ElastiCache — для тимчасового зберігання часто використовуваних даних у пам'яті з метою прискорення роботи системи.
     Для побудови мережевої інфраструктури в AWS використовуються мережеві сервіси. Amazon VPC дозволяє створити ізольоване мережеве середовище всередині хмарної платформи. Amazon Route 53 виконує функції DNS-сервісу, Amazon CloudFront прискорює доставку контенту користувачам через глобальну мережу, а Elastic Load Balancing автоматично розподіляє навантаження між серверами.
     Важливе місце в платформі займають сервіси безпеки. AWS IAM використовується для керування правами доступу користувачів і сервісів, AWS KMS – для шифрування даних, а AWS Shield і AWS WAF – для захисту від DDoS-атак та інших вебзагроз.
     Крім того, AWS містить засоби моніторингу та управління. Amazon CloudWatch допомагає відстежувати стан системи, збирати метрики та аналізувати журнали подій. AWS CloudTrail фіксує дії користувачів і звернення до API, а AWS Config дозволяє контролювати зміни в конфігурації ресурсів.
     3.4 Взаємодія компонентів
     Взаємодія між компонентами платформи AWS здійснюється через стандартизовані REST API та AWS SDK, доступні для більшості поширених мов програмування. Кожен сервіс публікує чітко визначений API, що дозволяє іншим сервісам та клієнтським застосункам взаємодіяти з ним незалежно від внутрішньої реалізації.
     Потоки даних в AWS організовані за такими принципами:
- синхронна взаємодія – клієнт звертається до сервісу та очікує відповіді. Застосовується для запитів до EC2, RDS, S3 та інших сервісів через HTTPS;
- асинхронна взаємодія – використовуються черги повідомлень (Amazon SQS), теми (Amazon SNS) та потоки подій (Amazon Kinesis, Amazon EventBridge) для відокремлення виробників і споживачів даних;
- подієво-орієнтована взаємодія – зміни стану ресурсів генерують події, що автоматично запускають обробники (наприклад, завантаження файлу в S3 може активувати функцію AWS Lambda).
     Типовий потік даних у вебзастосунку на AWS виглядає наступним чином: користувач надсилає запит через Amazon Route 53 (DNS), який направляє трафік до Amazon CloudFront (CDN), далі через Elastic Load Balancer (ELB) до Auto Scaling Group із EC2-інстансами або до API Gateway для безсерверної архітектури. Бізнес-логіка при цьому виконується на серверах, дані зберігаються в базах даних або файлових сховищах, а стан усієї системи відстежується через CloudWatch.
     3.5 Масштабованість та надійність
     Масштабованість є однією з ключових характеристик платформи AWS. Для забезпечення горизонтального масштабування використовується сервіс Auto Scaling, який автоматично додає або видаляє екземпляри EC2 залежно від визначених метрик (завантаження CPU, кількість запитів тощо). Elastic Load Balancing розподіляє трафік між активними інстансами, забезпечуючи рівномірне навантаження.
     Надійність платформи забезпечується через:
- зони доступності (Availability Zones) – фізично ізольовані центри обробки даних у межах одного регіону, з'єднані надшвидкісними мережевими каналами. Розміщення ресурсів у декількох зонах доступності усуває єдину точку відмови;
- реплікацію даних – більшість керованих сервісів (RDS Multi-AZ, DynamoDB, S3) автоматично реплікують дані між зонами доступності;
- кешування – Amazon ElastiCache та Amazon CloudFront зменшують навантаження на основні сервіси, знижують затримку та підвищують відмовостійкість;
- відновлення після збоїв – AWS надає інструменти для резервного копіювання (AWS Backup) та відновлення (Disaster Recovery) з чітко визначеними показниками RPO та RTO;
- контейнеризацію – використання Docker-контейнерів разом із ECS/EKS забезпечує ізоляцію застосунків та спрощує горизонтальне масштабування.
     Глобальна інфраструктура AWS охоплює понад 100 зон доступності та мережу точок присутності (PoP) Amazon CloudFront, що забезпечує мінімальну затримку для кінцевих користувачів по всьому світу.
     3.6 Приклади програмної реалізації архітектурних рішень
     Для демонстрації архітектурних принципів AWS наведено приклади взаємодії з ключовими сервісами платформи засобами AWS SDK для C# (AWSSDK.NET).
     Перший приклад демонструє завантаження файлу до Amazon S3 – об'єктного сховища AWS. Операція виконується через SDK-клієнт, що взаємодіє з REST API сервісу:
1 using Amazon.S3;
2 using Amazon.S3.Transfer;
3 
4 // Створення клієнта S3 та утиліти для завантаження файлів
5 var s3Client = new AmazonS3Client(Amazon.RegionEndpoint.USEast1);
6 var transferUtility = new TransferUtility(s3Client);
7 
8 static async Task UploadFileToS3Async(
9     string filePath, string bucketName, string objectKey)
10 {
11     // Асинхронне завантаження файлу до вказаного бакету
12     await transferUtility.UploadAsync(filePath, bucketName, objectKey);
13     // Підтвердження успішного завантаження
14     Console.WriteLine(
15         $"File {filePath} uploaded to s3://{bucketName}/{objectKey}");
16 }
17 
18 // Виклик функції: завантажуємо report.pdf до бакету my-bucket
19 await UploadFileToS3Async("report.pdf", "my-bucket", "reports/report.pdf");
     Другий приклад ілюструє безсерверну функцію AWS Lambda, що обробляє подію завантаження файлу в S3 та зберігає метадані до Amazon DynamoDB. Цей підхід демонструє подієво-орієнтовану взаємодію між сервісами:
1 using Amazon.Lambda.Core;
2 using Amazon.Lambda.S3Events;
3 using Amazon.DynamoDBv2;
4 using Amazon.DynamoDBv2.Model;
5 
6 public class Function
7 {
8     // Клієнт DynamoDB для збереження метаданих файлів
9     private readonly IAmazonDynamoDB _dynamoDb;
10 
11     public Function() =>
12         _dynamoDb = new AmazonDynamoDBClient();
13 
14     // Точка входу Lambda-функції: викликається при події завантаження в S3
15     public async Task<string> FunctionHandler(
16         S3Event s3Event, ILambdaContext context)
17     {
18         foreach (var record in s3Event.Records)
19         {
20             // Формування запиту із метаданими файлу для запису до DynamoDB
21             var request = new PutItemRequest
22             {
23                 TableName = "FileMetadata",
24                 Item = new Dictionary<string, AttributeValue>
25                 {
26                     ["FileKey"]    = new() { S = record.S3.Object.Key },
27                     ["Bucket"]     = new() { S = record.S3.Bucket.Name },
28                     ["Size"]       = new() { N = record.S3.Object.Size.ToString() },
29                     ["UploadedAt"] = new() { S = DateTime.UtcNow.ToString("o") }
30                 }
31             };
32             // Асинхронний запис метаданих до таблиці
33             await _dynamoDb.PutItemAsync(request);
34         }
35         return "Metadata saved.";
36     }
37 }
     Наведені приклади демонструють ключові архітектурні принципи AWS: слабке зв'язування між сервісами, подієво-орієнтовану взаємодію та використання керованих сервісів для зберігання даних.
     3.7 Переваги архітектурних рішень AWS
     Архітектура платформи AWS забезпечує низку суттєвих переваг порівняно з традиційними підходами до побудови програмних систем:
• Гнучкість і масштабованість – ресурси можуть бути виділені або вивільнені впродовж хвилин, що дозволяє ефективно реагувати на зміни навантаження;
• Висока доступність – розподіл ресурсів між зонами доступності та регіонами забезпечує доступність сервісів на рівні 99,99%;
• Безпека – AWS відповідає вимогам понад 90 стандартів безпеки та відповідності (ISO 27001, SOC 1/2/3, PCI DSS, HIPAA тощо);
• Зниження витрат – модель оплати за використання (Pay-as-you-go) усуває необхідність значних початкових капіталовкладень в обладнання;
• Прискорення інновацій – широкий спектр керованих сервісів дозволяє зосередитися на бізнес-логіці, а не на управлінні інфраструктурою;
• Глобальне охоплення – наявність регіонів по всьому світу дозволяє забезпечити низьку затримку для користувачів у будь-якій точці планети.


4 ВИСНОВКИ
     У процесі виконання практичного заняття було досліджено архітектуру хмарної платформи Amazon Web Services (AWS), яка є однією з найбільших та найскладніших розподілених програмних систем у сучасному ІТ-середовищі.
     Було встановлено, що платформа AWS побудована на основі сервісно-орієнтованої архітектури (SOA) з активним застосуванням мікросервісного підходу. Кожен сервіс AWS є незалежним компонентом зі своїм API, що забезпечує слабке зв'язування та можливість незалежного масштабування окремих частин системи.
     У ході роботи було визначено основні категорії сервісів AWS (обчислення, зберігання, бази даних, мережа, безпека, моніторинг) та проаналізовано механізми їх взаємодії. Встановлено, що AWS підтримує як синхронну, так і асинхронну та подієво-орієнтовану взаємодію між компонентами.
     Було досліджено механізми масштабованості та надійності платформи: використання зон доступності, автоматичне горизонтальне масштабування, реплікацію даних та кешування. Ці рішення забезпечують доступність сервісів на рівні 99,99% та можливість обслуговування мільярдів запитів на добу.
     Розглянуто практичні приклади взаємодії з сервісами AWS через SDK, що демонструють реалізацію архітектурних принципів платформи у програмному коді.
     Отримані результати підтверджують, що архітектура AWS є взірцевим прикладом реалізації принципів розподілених систем, мікросервісного підходу та хмарних обчислень, і може слугувати еталоном при проєктуванні сучасних масштабованих програмних систем.
